§05
AC_HealthComponent
Content Browser location
📁Content/ProjectAnomaly/Player/Components/
Lives in Player/Components/. Create it as an ActorComponent (not an Actor). Right-click in the Components/ folder → Blueprint Class → parent=ActorComponent. It will be added to BP_PlayerCharacter via the Components panel.
CREATE
Variables, dispatchers, TakeDamage function

Variables: MaxHealth (float=100, Instance Editable), CurrentHealth (float), bIsInvincible (bool). Event Dispatchers: OnDamageTaken (payload: DamageAmount float), OnDeath (no payload). BeginPlay: SET CurrentHealth=MaxHealth. Function TakeDamage(DamageInfo FS_DamageInfo): Branch bIsInvincible → exit. Break DamageInfo → subtract BaseDamage → Clamp(0, MaxHealth) → SET CurrentHealth → Call OnDamageTaken → Branch CurrentHealth≤0 → Call OnDeath. See the flowchart below.

Flow Summary
  • bIsInvincible gates all damage. Set it true during a dash or parry stagger to grant brief immunity.
  • Clamp prevents negative health. Always clamp before checking the death branch — never let health go below 0.
  • Both dispatchers are used. OnDamageTaken drives the HUD health bar and PP_HitFlash. OnDeath drives the death animation and death screen.
🔧 Debug
Tests
  • Add Print String in TakeDamage printing CurrentHealth — shoot player, confirm it decrements.
  • Add Print String "DEAD" in OnDeath — shoot until 0, confirm it fires exactly once.
Expected
  • CurrentHealth decrements by BaseDamage each shot.
  • OnDeath fires when CurrentHealth reaches 0, not before.
  • No damage applied when bIsInvincible=true.
§06
AC_RageComponent
Content Browser location
📁Content/ProjectAnomaly/Player/Components/
Lives in Player/Components/. This is the most critical system in the game — it blocks every ability unlock, every HUD binding, and the VFX pipeline. Build and test it before anything else that references rage state.
CRITICAL
Variables, AddRage, ConsumeRage, EndEnragedState

Variables: RageMax=100, RageCurrent=0, EnragedDuration=1.5, bIsEnraged=false, EnragedTimerHandle. Dispatchers: OnEnragedStateBegin, OnEnragedStateEnd (both no payload). Function AddRage(Amount float): Branch NOT bIsEnraged → SET RageCurrent=Clamp(RageCurrent+Amount, 0, RageMax) → Branch RageCurrent≥RageMax → ConsumeRage. ConsumeRage: SET bIsEnraged=true, RageCurrent=0 → Call OnEnragedStateBegin → Set Timer by Function Name "EndEnragedState" Time=EnragedDuration. Custom Event EndEnragedState: SET bIsEnraged=false → Call OnEnragedStateEnd.

Flow Summary
  • bIsEnraged gates AddRage. While enraged, no WP hits add rage — the meter stays at 0 until the state ends.
  • ConsumeRage resets to 0 before broadcasting. This means the HUD shows empty meter the moment enrage starts, not full.
  • The timer calls EndEnragedState as a Custom Event, not a function. This is intentional — it allows the timer to be cleared if needed in the future.
🔧 Debug
Tests
  • Print RageCurrent in AddRage — shoot WPs, confirm value increases.
  • Print "ENRAGE BEGIN" in OnEnragedStateBegin binding.
  • Print GetGameTimeSinceCreation in EndEnragedState — verify ~1.5s gap.
Expected
  • WP hits add rage; body shots do not.
  • OnEnragedStateBegin fires exactly once per fill.
  • EndEnragedState fires exactly 1.5s after begin.
  • Loop is clean: rage rebuilds from 0 after state ends.

PATTERN
AIC_ OnPossess — the standard wiring

Every AIC_ follows the same OnPossess pattern: Run Behavior Tree (BT_[Enemy]) → Cast GetControlledPawn to BP_Enemy_[Type] → GET AC_EnemyHealthComponent → Bind dispatchers to Blackboard writes: OnArmourBroken → Set BB Bool "bArmourBroken"=true; OnStunBegin → "bIsStunned"=true; OnStunEnd → false; OnDeath → "bIsDead"=true. ℹ️
Finish Execute is mandatory. Every BTT_ (Behavior Tree Task) must call Finish Execute on ALL code paths — including cast failures and early exits. If Finish Execute is never called, the BT freezes on that node permanently. This is the most common BT bug.

Navigate to Enemies/Shared_AI/. Create all shared nodes here so all enemy BTs can reference the same assets. §20
Full Rage Loop — end-to-end
Content Browser location
📁Content/ProjectAnomaly/Player/Components/
This section covers how all systems wire together in BP_PlayerCharacter's BeginPlay. No new assets are created here — it is the binding pass that connects the rage dispatchers to the 4 ability unlocks and 3 ability reverts.
WIRE
BP_PlayerCharacter BeginPlay — 7 rage bindings via Sequence node

In BP_PlayerCharacter BeginPlay, use a Sequence node. GET RageComponent. Bind OnEnragedStateBegin to 4 Custom Events: WeaponBoost (SET WeaponComponent.bNextShotBoosted=true), UnlockDash (SET MovementExtComponent.bDashAvailable=true), WidenParry (ParryComponent.SetWindowLeniency(2.0)), EnableSkill (SET SkillSlotComponent.bSkillAvailable=true). Bind OnEnragedStateEnd to 3: LockDash (=false), ResetParry (SetWindowLeniency(1.0)), DisableSkill (=false).

Flow Summary — Complete Rage Loop
  • WP hit → AddRage → meter fills → ConsumeRage sets bIsEnraged=true and calls OnEnragedStateBegin.
  • 4 ability unlocks fire simultaneously from the single OnEnragedStateBegin dispatcher broadcast.
  • 1.5s timer → EndEnragedState sets bIsEnraged=false and calls OnEnragedStateEnd.
  • 3 ability reverts fire simultaneously from OnEnragedStateEnd.
  • Meter rebuilds from 0. The loop can repeat indefinitely.
