Naming Convention & Abbreviation Index
All assets follow UE5 community naming conventions. Every prefix is listed below. When adding new assets, always pick the matching prefix — this keeps the Content Browser filterable and the architecture self-documenting.
BP_
BlueprintActor, Character, Pawn, or utility Blueprint
AC_
ActorComponentModular component attached to an actor
BPI_
Blueprint InterfaceCross-system communication contract
DA_
Data AssetPrimary Data Asset — designer-tunable values
DT_
DataTableTable of struct rows (CSV-style)
FS_
Struct (FStruct)Custom data structure used across systems
WBP_
Widget BlueprintUMG UI widget
BT_
Behavior TreeAI behavior tree asset
BB_
BlackboardAI Blackboard data asset
BTT_
BT TaskBehavior Tree leaf task node
BTS_
BT ServiceBehavior Tree polling service node
BTD_
BT DecoratorBehavior Tree condition/gate node

Enemy AI — Components & Blackboard Keys
Shared components live on BP_EnemyBase and are inherited by every enemy. Unique components are added only on the specific child BP that needs them. All Behavior Trees read/write only their own Blackboard — they never call other Blueprints directly.
Shared — all enemies
AC_EnemyHealthComponent
Separate health + armour pools. Armour reaches 0 first before health takes full damage. Broadcasts OnArmourBroken and OnDeath so the AI Controller can trigger BT state changes.
  • float MaxHealth, CurrentHealth float
  • float MaxArmour, CurrentArmour float
  • bool bArmourBroken bool
  • Event OnArmourBroken delegate
  • Event OnDeath (Killer) delegate
  • Func TakeDamage (FS_DamageInfo) callable
Shared — all enemies
AC_WeakPointComponent
Manages N collision volumes parented to enemy sockets. Confirms a WP hit on overlap+damage. Calls BPI_Rageable::NotifyWeakPointHit on the player to feed rage — the component doesn't know anything else about the player.
  • Array<BP_WeakPointActor> WeakPoints array
  • Array<bool> bWPExposed array
  • int ActiveWPCount int
  • float RageAmountPerHit float
  • Event OnWeakPointHit (Index, Damage) delegate
  • Func GetExposedCount () → int callable
Shared — all stun-capable enemies
AC_StunComponent
Accumulates stun damage from sustained fire. When the threshold (from DA_EnemyData) is crossed, sets bIsStunned and starts recovery timer. BTD_IsStunned decorator pauses the Behavior Tree during stun.
  • float StunAccumulator float
  • float StunThreshold (from DA_EnemyData) float
  • float StunDuration float
  • bool bIsStunned bool
  • Event OnStunBegin, OnStunEnd delegate
  • Func AddStunDamage (float Amount) callable
Shared — all enemies (UE Native)
AIPerceptionComponent
Configured on BP_EnemyBase. Sight detects the player within radius and half-angle defined per enemy in DA_EnemyData. On detection, AIC_ writes the player reference to the Blackboard TargetActor key.
  • UAISenseConfig_Sight SightConfig UE native
  • UAISenseConfig_Hearing HearingConfig UE native
  • float SightRadius (per DA_EnemyData row) float
  • float LoseSightRadius float
  • Event OnPerceptionUpdated → AIC_ writes BB delegate
Brute — BB_Brute keys
Blackboard Keys
All runtime AI state for the Brute. BT tasks read and write these — no direct Blueprint calls to any external system.
  • TargetActor Object (AActor)
  • TargetLocation Vector
  • bHasLOS Bool
  • bIsStunned Bool
  • bArmourBroken Bool
  • AttackCooldownReady Bool
Flying Eye — BB_FlyingEye keys
Blackboard Keys
bEyeDestroyed is set by AC_WeakPointComponent when WP index 0 (the eye) is confirmed hit — movement tasks check this flag to stop strafing.
  • TargetActor Object (AActor)
  • StrafeDestination Vector (EQS result)
  • bHasLOS Bool
  • bEyeDestroyed Bool
  • bImmobilised Bool
  • FireCooldownReady Bool
Spawner — BB_Spawner keys
Blackboard Keys
RetreatDestination is populated by EQS_Spawner_Retreat. SpawnBudget is managed by AC_SpawnDirectorComponent — decremented on spawn, incremented on fodder death.
  • TargetActor Object (AActor)
  • RetreatDestination Vector (EQS result)
  • SpawnBudget Int
  • bCanSpawn Bool
  • bIsRetreating Bool
Spawner only — unique component
AC_SpawnDirectorComponent
Manages a spawn budget to prevent infinite fodder. Decrements on spawn; refills when fodder dies via binding to OnDeath delegate. Writes bCanSpawn to Blackboard.
  • int MaxSpawnBudget, CurrentBudget int
  • float SpawnInterval float
  • TArray SpawnPointRefs array
  • Func TrySpawnFodder () callable
  • Func OnFodderDied () bound to OnDeath

Player Actor Components
ActorComponent — Player
AC_HealthComponent
Single source of truth for player HP. Broadcasts events; never calls other systems directly.
  • float CurrentHealth, MaxHealth float
  • bool bIsInvincible bool
  • Event OnDamageTaken (Amount) delegate
  • Event OnDeath delegate
  • Func TakeDamage (FS_DamageInfo) callable
  • Func Heal (float Amount) callable
ActorComponent — Player
AC_RageComponent
Owns the Rage meter. On meter full, starts Enraged State and broadcasts to all listeners. Timer fires OnEnragedStateEnd — all listeners revert state independently.
  • float RageCurrent, RageMax float
  • float EnragedDuration (1.5s default) float
  • bool bIsEnraged bool
  • Event OnEnragedStateBegin delegate
  • Event OnEnragedStateEnd delegate
  • Func AddRage (float), ConsumeRage () callable
ActorComponent — Player
AC_WeaponComponent
Manages three-slot inventory. Weapons are ChildActorComponents. Delegates Fire() to active slot. Holds bNextShotBoosted flag set by OnEnragedStateBegin.
  • Array<BP_WeaponBase> WeaponSlots [3] array
  • int ActiveWeaponIndex int
  • bool bNextShotBoosted bool
  • float NextShotMultiplier (3.0×) float
  • Event OnWeaponSwitched, OnAmmoChanged delegate
  • Func Fire (), SwitchWeapon (Index) callable
ActorComponent — Player
AC_MovementExtComponent
Extends CMC with grapple (always available) and rage-gated dash. Checks BPI_Rageable::IsEnraged() on owner — no direct reference to AC_RageComponent.
  • BP_GrappleAnchor CurrentAnchor soft ref
  • float GrappleSpeed, GrappleRange float
  • float DashForce, DashDuration float
  • bool bDashAvailable (rage-gated) bool
  • Event OnGrappleAttached, OnDashUsed delegate
  • Func TryGrapple (), TryDash (Direction) callable
ActorComponent — Player
AC_ParryComponent
Parry input, window timing, perfect parry detection. Window leniency widened by OnEnragedStateBegin event binding — decoupled from AC_RageComponent.
  • float BaseParryWindow, CurrentParryWindow float
  • bool bParryActive bool
  • Func ActivateParry (), DeactivateParry () callable
  • Func SetWindowLeniency (Multiplier) callable
  • Event OnParrySuccess (bIsPerfect) delegate
  • Event OnPerfectParry (DeflectedActor) delegate
ActorComponent — Player
AC_UtilityComponent
Satchel Charge: throw, 2s fuse, detonation, radial shockwave. URadialForceComponent handles physics push internally. 45s cooldown gates re-throw.
  • float Cooldown (45s), CooldownRemaining float
  • float DetonationDelay (2s) float
  • float ShockwaveRadius, ShockwaveForce float
  • bool bCanThrow bool
  • Event OnSatchelThrown, OnDetonated, OnCooldownComplete delegate
  • Func ThrowSatchel (Dir), DetonateSatchel () callable
ActorComponent — Player
AC_SkillSlotComponent
Registry and executor for the Special Skill slot. Holds a soft ref to equipped DA_SkillData. EquipSkill() is the only entry point for swapping — called from BP_SavePoint interaction. Persists selection via BP_Save_Anomaly.
  • DA_SkillData EquippedSkill soft ref
  • TArray<DA_SkillData> UnlockedSkills array
  • bool bSkillAvailable (rage-gated) bool
  • Event OnSkillEquipped, OnSkillActivated delegate
  • Func ActivateSkill (), EquipSkill (DA_SkillData) callable
  • Func GetAvailableSkills () → Array callable

Blueprint Interfaces
Rule: No system holds a direct typed reference to another system's class. All cross-actor calls go through an interface. Any implementor can be replaced or removed without touching the caller.
BPI_Damageable — Player, all enemies
FunctionParametersPurpose
ReceiveDamageFS_DamageInfoUniversal damage entry. Struct carries amount, type, WP flag, armour flag, instigator.
IsAlive→ boolSafe liveness check before applying logic
GetCurrentHealth→ floatRead-only query for UI and AI
BPI_WeakPointTarget — All enemies with weak points
FunctionParametersPurpose
RegisterWeakPointHitint Index, float DamageCalled by AC_WeakPointComponent on confirmed hit
GetWeakPointCount→ intQueried by UI and arena manager
IsWeakPointExposedint Index → boolChecks if armour stripped from this WP (Brute)
BPI_Rageable — Player only
FunctionParametersPurpose
IsEnraged→ boolPolled by AC_MovementExtComponent and AC_SkillSlotComponent to gate abilities
GetRagePercent→ float 0–1WBP_RageMeter UI binding
NotifyWeakPointHitfloat RageAmountWeapons feed rage without touching AC_RageComponent directly
BPI_Parryable — BP_Projectile_Base children
FunctionParametersPurpose
CanBeParried→ boolProjectile reports if currently in parry-eligible (highlighted) state
OnParriedAPawn* ParryingPawnProjectile handles redirection or destruction
OnPerfectParriedAPawn* ParryingPawnTriggers stagger on source enemy via AC_StunComponent
BPI_Interactable — BP_SavePoint, future interactables
FunctionParametersPurpose
InteractAPawn* InteractorSavePoint opens Skill menu + saves; future objects do their own thing
GetInteractPrompt→ FTextLocalisation-safe prompt string for WBP_HUD
CanInteractAPawn* → boolGate — e.g. disallow skill swap mid-combat
BPI_SkillEquippable — BP_SkillBase children
FunctionParametersPurpose
ExecuteAPawn* InstigatorTriggers effect — called by AC_SkillSlotComponent only
GetSkillData→ DA_SkillDataReturns Data Asset for WBP_SkillSelectMenu display
IsReadyToUse→ boolPer-skill cooldown check (future skills may have own cooldowns)

Animation Pipeline
Every character and enemy needs its own Anim Blueprint, at minimum one Blend Space for locomotion, montages for actions, and Anim Notifies to fire events at specific frames. The ABP_ is the only place where animation state lives — Blueprints never manually set animation poses.
Anim Blueprint — Player
ABP_Player
Controls the player's animation state machine. Reads speed, direction, bIsGrounded, bIsGrappling from the Character Movement Component each tick and drives the blend space. Montages layer on top for actions.
  • float Speed ← CharacterMovement.Velocity.Size float
  • float Direction ← Calculate Direction() float
  • bool bIsGrounded ← CharacterMovement.IsMovingOnGround bool
  • bool bIsGrappling ← AC_MovementExtComponent.bGrappling bool
  • bool bIsEnraged ← BPI_Rageable.IsEnraged on Owner bool
  • BS_Player_Locomotion — primary state machine blend blend space
Anim Blueprint — Per Enemy
ABP_[Enemy] (one per enemy type)
Each enemy has its own ABP_ reading from its own variables. The ABP_ never reads from Blueprint components directly — the AIC_ or the enemy BP pushes state flags into variables the ABP_ can read safely.
  • float Speed ← CharacterMovement.Velocity.Size float
  • bool bIsStunned ← set by AIC_ when BB bIsStunned changes bool
  • bool bIsDead ← set by AC_EnemyHealthComponent.OnDeath bool
  • BS_[Enemy]_Locomotion — idle/walk/run blend blend space
  • Slot "DefaultSlot" — montages play into this slot slot
Anim Notify — Point Event
AN_[Enemy]_AttackHit
A point-in-time notify placed on the attack montage at the exact frame the weapon makes contact. BTT_AttackMelee binds to On Notify and applies damage only at this moment — not at the start of the swing.
  • Placed on AM_[Enemy]_Attack at impact frame placement
  • BTT_AttackMelee binds to OnPlayMontageNotifyBegin binding
  • Notify Name must match exactly in task BP string match
Anim Notify State — Duration Window
ANS_[Enemy]_AttackWindow
A duration-based notify marking the full window of an attack during which the player could theoretically parry. The parry system checks whether this window is active on the projectile's source enemy. Different from AN_ — has Begin and End events.
  • Placed on AM_[Enemy]_Attack spanning wind-up to impact placement
  • NotifyBegin → sets bAttackWindowOpen=true on enemy event
  • NotifyEnd → sets bAttackWindowOpen=false event
Blend Space
BS_[Character]_Locomotion
A 2D blend space with Speed on the X axis (0=idle, 600=run) and Direction on the Y axis (-180 to 180 for strafing). The ABP_ drives this every tick. Each character has its own with appropriate animation sets.
  • Axis X: Speed (0–600) float axis
  • Axis Y: Direction (-180 to 180) float axis
  • Samples: Idle, WalkFwd, WalkBwd, WalkL, WalkR, RunFwd anim samples
IK Rig + IK Retargeter
IKR_Player / IKRT_EnemyToPlayer
IK Rig defines the solver chains (foot IK, hand IK on weapon grip) for the player skeleton. IK Retargeter maps animations from one skeleton to another — used if enemies share a base rig with the player or if purchased animations need remapping.
  • IKR_Player — foot IK chains, hand IK for weapon IK Rig
  • IKRT_EnemyToPlayer — remap enemy anims to player rig IK Retargeter
  • Foot IK in ABP_: Two Bone IK node per leg ABP node
// 04
Footstep System
Footsteps are driven by Anim Notifies in the locomotion animations, not by timers. The notify fires a line trace downward at the foot bone, reads the physical material, and picks the correct sound from DT_FootstepSurfaces. This applies to both the player and enemies.
Anim Notify — Footstep
AN_Player_FootstepL / AN_Player_FootstepR
Placed on locomotion animations at the exact frame each foot strikes the ground. Fires a line trace from the foot bone downward to read the physical material, then plays the matching sound from DT_FootstepSurfaces and spawns NS_Footstep_Dust.
  • Line Trace from foot bone → 20 units down trace
  • Hit Physical Material → row lookup in DT_FootstepSurfaces data lookup
  • Play Sound at Location ← matched SC_ asset audio
  • Spawn NS_Footstep_Dust at hit location VFX
DataTable
DT_FootstepSurfaces
Maps physical surface types to footstep sound cues. Adding a new surface type (e.g. metal grating) means adding one row — no notify changes needed. Physical Materials must be assigned to level geometry for this to work.
  • Column: SurfaceType (EPhysicalSurface enum) enum
  • Column: FootstepSound (SC_ soft ref) soft ref
  • Column: FootstepVFX (NS_ soft ref, optional) soft ref
  • Rows: Default, Metal, Concrete, Gravel, Water rows
// 05
Hit Reactions + Death Ragdoll
Hit reactions are short montages that interrupt the current animation when an enemy takes significant damage. Death transitions the enemy from animation to physics ragdoll. Both are triggered from AC_EnemyHealthComponent events.
System — Hit Reaction
AM_[Enemy]_HitReact (directional variants)
Short (~0.3s) interrupt montages played when an enemy takes a hit. Directional variants (Front, Back, Left, Right) are chosen by calculating the hit direction relative to the enemy's forward vector. Triggered from AC_EnemyHealthComponent.OnDamageTaken.
  • AM_[Enemy]_HitReact_Front/Back/Left/Right montages
  • Triggered by: AC_EnemyHealthComponent.OnDamageTaken trigger
  • Direction calc: Dot(HitDir, EnemyForward) → choose montage logic
  • Blend In: 0.1s — quick interrupt, Blend Out: 0.2s blend settings
  • Montage Section: overrides locomotion briefly then blends back slot
System — Death Ragdoll
PHYS_[Enemy] + Death transition
On OnDeath, the enemy plays AM_[Enemy]_Death briefly then transitions to physics simulation. The PHYS_ asset defines which bones are simulated. The capsule collision is disabled so the corpse doesn't block movement.
  • Trigger: AC_EnemyHealthComponent.OnDeath → delay 0.2s trigger
  • Set All Bodies Simulate Physics = true on SK Mesh node
  • Set Collision Profile = Ragdoll on SK Mesh node
  • Capsule: Set Collision Enabled = No Collision node
  • Apply Last Velocity impulse to ragdoll root node
  • Dissolve via MF_Dissolve after 5s → Destroy Actor cleanup
// 06
Post Process + Rendering
Post Process effects are driven by a Post Process Volume placed in each level, with material parameters animated at runtime. The Enrage effect and hit flash are both post process materials triggered from the rage/damage event systems.
Post Process Material
PP_Enrage
Full-screen colour shift during Enraged State. A scalar parameter "EnrageIntensity" is animated from 0→1 on OnEnragedStateBegin and back to 0 on OnEnragedStateEnd using a Timeline in BP_PlayerCharacter. Add this material to the Post Process Volume's Materials array.
  • Scalar param: EnrageIntensity (0–1) parameter
  • Effect: red vignette + slight desaturation of world visual
  • Driven by: Timeline in BP_PlayerCharacter rage bindings driver
  • Set Scalar Parameter Value on PostProcessComponent node
Post Process Material
PP_HitFlash
Brief white flash on player taking damage. Driven by a short curve in a Timeline triggered from AC_HealthComponent.OnDamageTaken binding in BP_PlayerCharacter. Intensity spikes to 1.0 then falls to 0 over 0.15s.
  • Scalar param: HitFlashIntensity (0–1) parameter
  • Duration: 0.15s curve — spike then decay timing
  • Driven by: OnDamage_HitFlash Custom Event in BP_PlayerCharacter driver
World Actor — Per Level
PostProcessVolume (placed in level)
One Post Process Volume set to Infinite Extent covers the whole level. Holds base colour grading, bloom, ambient occlusion, and the PP_ material slots. Only one PPV needed unless different zones need different looks.
  • Infinite Extent (Unbound): true setting
  • Materials array: PP_Enrage, PP_HitFlash materials
  • Bloom: Intensity=0.5, Threshold=-1 bloom
  • Ambient Occlusion: Intensity=0.5, Radius=20 AO
  • Color Grading: LUT assigned per act/location LUT
// 07
Enhanced Input System
UE5 uses Enhanced Input. Every action is an IA_ asset. The IMC_ binds IA_ assets to hardware keys. This decouples actions from keys completely — remapping controls means editing IMC_ only, no Blueprint changes.
Input Mapping Context
IMC_Default
The only IMC needed for standard gameplay. Added to the player's Enhanced Input subsystem in BP_PlayerController BeginPlay via Add Mapping Context. Contains all key-to-action bindings for keyboard/mouse and gamepad.
  • IA_Fire → Left Mouse Button / RT binding
  • IA_Parry → Right Mouse Button / LT binding
  • IA_Grapple → E / RB binding
  • IA_Dash → Shift / B binding
  • IA_Utility → Q / LB binding
  • IA_Skill → F / Y binding
  • IA_WeaponWheel → Tab / Left Stick Click (hold) binding
  • IA_Move → WASD / Left Stick Axis binding
  • IA_Look → Mouse XY / Right Stick Axis binding
  • IA_Jump → Space / Face Button Bottom binding
Setup Note
Registering IMC_ in BP_PlayerController
Add this to BP_PlayerController BeginPlay — without it, no Enhanced Input actions fire at all.
  • Get Enhanced Input Subsystem (from PlayerController) node
  • Add Mapping Context → IMC_Default, Priority=0 node
  • Priority 0 = default gameplay context note
  • Add a second IMC at Priority 1 for UI/menu states tip

Missing UI Screens
WBP_PauseMenu, WBP_DeathScreen, and WBP_LoadingScreen complete the player-facing UI. These are created/destroyed by BP_PlayerController or the Game Mode — they are never open simultaneously.
Widget Blueprint
WBP_PauseMenu
Opened on Escape/Start. Sets game to paused state. Contains Resume, Settings (stub), and Quit buttons. Opened by BP_PlayerController at ZOrder=10 so it appears above all other UI.
  • Input: IA_Pause → BP_PlayerController → Create/Remove widget trigger
  • Set Game Paused = true on open node
  • Set Input Mode UI Only on open node
  • Resume button → Set Game Paused=false, remove widget, restore game input button
  • Quit button → Quit Game node button
Widget Blueprint
WBP_DeathScreen
Shown on AC_HealthComponent.OnDeath in BP_PlayerCharacter. After a 2s delay (death animation plays), fades in. Offers Respawn (reload level) and Main Menu. Prevents input to the character while visible.
  • Trigger: AC_HealthComponent.OnDeath → 2s timer → show widget trigger
  • Fade in via UMG animation on construct animation
  • Respawn button → Open Level (current level name) button
  • Main Menu button → Open Level "L_MainMenu" button
Widget Blueprint
WBP_LoadingScreen
Shown during level transitions. Created just before Open Level is called and removed after the new level's BeginPlay fires. Contains a looping animation or tip text. BP_GI_Anomaly creates and owns this since it persists across levels.
  • Created by BP_GI_Anomaly before any Open Level call owner
  • Removed in the new level's BP_GM_Anomaly BeginPlay removal
  • Looping spinner or progress hint animation content
// 09
Scalability Rules for Future GDD Changes
+W
Adding a new weapon

Create a child BP of BP_WeaponBase. Add one row to DT_Weapons. Override Fire() for unique behaviour. Zero changes to existing Blueprints.

+S
Adding a new Special Skill

Create a child BP of BP_SkillBase, implement BPI_SkillEquippable::Execute(). Add a row to DT_Skills. Auto-appears in WBP_SkillSelectMenu — no UI code changes.

+E
Adding a new enemy type

Child BP of BP_EnemyBase + row in DT_Enemies + new BT_, BB_, and AIC_. Add unique components as needed. Nothing else changes.

+M
Adding a new movement ability

Add a function to AC_MovementExtComponent. Gate with BPI_Rageable::IsEnraged() if needed. Bind input in BP_PlayerController. No existing movement code touched.

≠
Changing any tunable value

Edit only the relevant DT_ row. Covers weapon damage, rage meter size, enraged duration, satchel cooldown, parry windows, enemy health, armour, stun thresholds — no Blueprint graphs opened.

∅
Removing a system entirely

Interface/event-driven design means removing e.g. AC_ParryComponent + BPI_Parryable leaves no dangling hard references in any other Blueprint.
